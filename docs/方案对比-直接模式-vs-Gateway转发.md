# 两种工作模式对比：直接模式 vs Gateway 转发

> 本文用日常生活中的比喻来解释 TDAI MCP Server 的两种工作模式，
> 方便非技术背景的同学理解它们的区别。

---

## 一句话概括

- **直接模式** = 你用自己的笔记本做笔记，简单够用
- **Gateway 转发** = 你请了一个专业秘书，帮你整理笔记、提炼重点、记住你的习惯

---

## 模式 A：直接模式（Direct Mode）

### 它是怎么工作的？

```
你  ──提问──▶  MCP Server (Python)  ──写入──▶  SQLite 数据库
                                    ──读取──▶
```

就像一个**个人笔记本**：

- 你说的话、Agent 的回答，直接记在笔记本里
- 下次你想查之前说过什么，Server 就翻笔记本找关键词
- 所有操作都是"直来直去"的

### 优点

- ✅ **不需要安装 Node.js**（少装一个软件）
- ✅ 纯 Python，环境简单
- ✅ 启动快，占内存少

### 缺点

- ❌ 搜索靠"关键词匹配"——就像在笔记本里按 Ctrl+F 搜关键词，搜"苹果"就找不到"iPhone"
- ❌ 不会自动提炼总结——你说 100 句话，它就记 100 句，不会帮你归纳成"这个用户喜欢简洁的回答"
- ❌ 没有"画像"——它不记得你的使用习惯和偏好

### 适合谁用？

- 只是想**尝鲜试用**的人
- 电脑上没有 Node.js 环境
- 对记忆要求不高，能搜到之前说过的话就行

---

## 模式 B：Gateway 转发（Gateway Mode）

### 它是怎么工作的？

```
你  ──提问──▶  MCP Server (Python)  ──HTTP──▶  TDAI Gateway (Node.js)
                                  ◀──转发────  │
                                               ├── 智能搜索引擎
                                               ├── 记忆提炼工厂
                                               └── 用户画像生成器
```

就像一个**专业秘书团队**：

- 你说的话，**秘书（Gateway）** 先收下
- 秘书不只是记录，还会做三件事：
  1. **智能搜索**——不光找关键词，还能理解意思，搜"苹果"也能找到"iPhone"
  2. **记忆提炼**——把你的 100 句话归纳成几条核心要点
  3. **建立画像**——慢慢了解你的说话风格、常用工具、工作习惯

### 优点

- ✅ **智能搜索**——BM25 + 向量 + RRF 三重检索（简单说：既找关键词，也理解语义，最后把两种结果综合排序）
- ✅ **自动提炼**——对话到一定数量后，自动生成"场景摘要"和"用户画像"
- ✅ **去重**——同样的信息不会重复记录
- ✅ **上下文压缩**——长任务中可以生成 Mermaid 流程图，把几十万 Token 的日志压缩成几百 Token 的符号图
- ✅ **空闲时自动工作**——你停下来的时候，秘书在后台默默整理笔记

### 缺点

- ❌ 需要安装 Node.js（额外装一个运行环境）
- ❌ 多一个后台进程，占用多一点内存
- ❌ 启动稍慢几秒（Gateway 需要初始化）

### 适合谁用？

- 希望 Agent **越用越懂你**的人
- 日常使用频率高、对话量大的用户
- 愿意花几分钟配置，换取长期更好的体验

## 一张表看懂区别

| 对比项 | 直接模式 | Gateway 转发 |
|--------|---------|-------------|
| **需要安装什么** | 只需 Python | Python + Node.js |
| **启动速度** | 立即启动 | 慢 3-5 秒 |
| **搜索方式** | 关键词匹配（Ctrl+F） | 语义理解 + 关键词混合搜索 |
| **记忆提炼** | ❌ 不会 | ✅ 自动归纳总结 |
| **用户画像** | ❌ 不会 | ✅ 自动建立 |
| **重复内容处理** | ❌ 会重复存 | ✅ 自动去重 |
| **长任务压缩** | ❌ 不支持 | ✅ 生成 Mermaid 符号图 |
| **记忆准确度** | 一般 | 高（论文数据：+59%） |
| **适合人群** | 尝鲜、轻量使用 | 重度用户、追求体验 |

## 论文数据怎么说？

TencentDB Agent Memory 的公开测试数据（基于 OpenClaw 插件）：

| 能力 | 不加记忆 | 加记忆（Gateway 模式） | 提升 |
|------|---------|----------------------|------|
| 搜索类任务成功率 | 33% | **50%** | **+51%** |
| 编程任务成功率 | 58.4% | **64.2%** | **+10%** |
| Token 消耗（搜索） | 221M | **85M** | **节省 61%** |
| 用户画像准确率 | 48% | **76%** | **+59%** |

> 注：直接模式因为没有完整的 pipeline，效果介于"不加记忆"和"Gateway 模式"之间。

## 实际体验对比

### 场景：你早上说过"我喜欢用 Python 写脚本"

**直接模式**：
```
你：帮我写个脚本
Agent：好的（它不记得你喜欢 Python，可能给你生成 Bash 脚本）
你查记忆：搜索"喜欢 Python"
→ 找到那条记录，但 Agent 不会主动利用它
```

**Gateway 模式**：
```
你：帮我写个脚本
Agent：好的（画像里有"用户偏好Python"，自动生成 Python 脚本）
你不需要查记忆，Agent 已经主动用上了
```

### 场景：你在做一个复杂的项目，已经聊了 200 轮

**直接模式**：
```
上下文越来越长，Agent 开始"忘记"前面说了什么
响应速度变慢，Token 消耗越来越大
```

**Gateway 模式**：
```
中间步骤被压缩成 Mermaid 流程图（几百 Token）
Agent 上下文保持轻量
需要查细节时，按 node_id 瞬间找回完整原文
```

## 如何选择？

```
你想试试这个工具？
├── 是 ──▶ 电脑有 Node.js 吗？
│           ├── 有 ──▶ Gateway 模式（推荐，体验最好）
│           └── 没有 ──▶ 直接模式（先体验，以后随时可以升级）
│
└── 不想试 ──▶ （那就不用纠结了 😄）
```

**随时可以切换**：两种模式共用同一个 SQLite 数据库。你今天用直接模式记了一堆对话，明天切到 Gateway 模式，之前的记录全都在，Gateway 还能帮你重新提炼总结。
